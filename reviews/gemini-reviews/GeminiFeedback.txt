Here is a thorough review of your evaluation pipeline, addressing each of your specific questions and highlighting a few critical, subtle bugs that could compromise your metrics.
1. Is BCF applied exactly once in every code path?

Yes, BCF is applied exactly once in all paths.

    Zero-shot: Applied once in predict_holdout via inverse_transform(...) * bcf. When evaluated in the adaptation loop for N=0, adapt_none computes it fresh from y_pred_ms, applying it exactly once.

    Bayesian / None: Both apply the global BCF exactly once after inverse transformation.

    Old 2-param: The global BCF is passed in but ignored. Instead, it computes bcf_trial and applies it exactly once.

    OLS log-log: Completely bypasses the CatBoost logic and BCF entirely, predicting directly in native space.

2. Is the Bayesian adaptation mathematically correct?

Yes, with one minor caveat.
Your formula, δ=N+kN​⋅mean(resid), correctly implements Bayesian shrinkage for a mean offset. The logic correctly calculates the residual as ytrue​−ypred​ and adds it back to the test predictions.

    The Edge Case: If N=0, δ=0, which cleanly falls back to the zero-shot prediction. However, if N+k=0 (e.g., if you ever tested a negative k for some reason), it would divide by zero. Given k=30 is your default, this is mathematically sound.

3. Is the old 2-parameter adaptation correct?

No, there are two severe mathematical flaws in adapt_old_2param.

    Flaw 1: Slope clipping breaks the intercept.
    You fit the OLS and retrieve intercept b and slope a. Then, you clip a: a = np.clip(a, 0.1, 10.0). If the slope is clipped, the line no longer passes through the centroid of your calibration data. This introduces a massive, unintended bias shift.
    Fix: If you clip a, you must recalculate b using the calibration centroids: b=yˉ​−axˉ.

    Flaw 2: The N=1 collapse.
    When N=1, you calculate a perfect offset: b = float(cal_true[0] - cal_pred[0]) with a=1.0. Because of this, corrected_ms[cal_idx] will exactly equal cal_true. When you calculate bcf_trial, you are dividing np.mean(cal_true_native) by itself, guaranteeing that bcf_trial will always be exactly 1.0. The per-trial BCF calculation silently neuters itself for N=1.

4. Is R² computed correctly everywhere?

Yes.

    In compute_site_metrics, you calculate ss_tot = np.sum((y_true - y_true.mean()) ** 2). Because this function receives different slices of data depending on the context, y_true.mean() correctly adapts.

    For per-site metrics, it uses the site-level mean.

    For pooled metrics, it passes the fully concatenated arrays, meaning it uses the global mean.

    For adaptation trials, it uses the local mean of the test slice. If the test slice has near-zero variance, ss_tot < 1e-10 catches it and correctly returns NaN, preventing the extreme negative R² distortions you mentioned.

5. Is the random seeding truly deterministic?

No. This is a critical bug.
Your seed formula is seed + hash(site_id) % (2**31) + n_val * 1000 + trial.
Since Python 3.3, the built-in hash() function for strings and bytes is salted with a random integer generated at Python startup (to prevent dictionary DoS attacks). If site_id is a string (e.g., "USGS-12345"), hash(site_id) will return a completely different number every time you execute the script.
Fix: Use hashlib to ensure cross-session determinism:
Python

import hashlib
site_hash = int(hashlib.md5(str(site_id).encode()).hexdigest(), 16) % (2**31)
rng = np.random.default_rng(seed + site_hash + n_val * 1000 + trial)

6. Subtle Bugs and Pitfalls

    NaN generation in OLS log-log: In adapt_ols_loglog, if valid.sum() == 0, you return np.full(len(test_turb), np.nanmedian(cal_ssc)). If there are no valid values in cal_ssc (e.g., all were zero/negative), np.nanmedian will raise a runtime warning and return NaN. This NaN will propagate through your predictions.

    Feature Imputation Silence: In predict_holdout, you fill missing numeric features with train_medians[col]. If a column has NaNs but was accidentally omitted from the train_medians dictionary in your JSON, fillna will do nothing, and the NaN will be passed silently to CatBoost. CatBoost can handle NaNs, but if this is unintended, it will mask a data pipeline failure.

    List Comprehension Median Warnings: In run_adaptation_curve, if a specific site/N combo results entirely in NaN R² values across all 50 trials, valid_trials will be empty. The if valid_trials: check handles this perfectly, but if you ever switch to calculating array-wise medians directly without the list comprehension filter, np.nanmedian on an all-NaN array will throw warnings.

7. What to change if building from scratch

    Abstract the Transforms: Instead of passing transform_type and lmbda as raw strings/floats to every adaptation function, create a TargetTransformer class. This centralizes forward(), inverse(), and safe-clipping logic.

    Empirical Bayes for k: Your Bayesian shrinkage uses a hardcoded k=30. In a truly optimal Bayesian setup, k should be derived from the ratio of the global residual variance to the site-specific residual variance. Sites with highly erratic sensors should have a higher k (more shrinkage) than highly stable sites.

    Decouple Evaluation from Prediction: run_adaptation_curve runs the model predictions and the adaptation logic and the metric computation simultaneously. It would be cleaner to have the adaptation functions return purely the corrected arrays, store those in a structured format (like an xarray Dataset or a multi-index DataFrame), and run a single generic metric-computation pass at the very end.

    Critical Methodological Trap: The Anchor Site Data Leak

    The Issue: The 50 "Anchor Sites" were identified by finding which training sites consistently appeared in the random subsets that scored highest on the 76 holdout sites. Evaluating the final Anchor-50 model on those exact same 76 holdout sites constitutes severe data leakage. The holdout set was inadvertently used to select the training data.

    The Fix: You must use K-Fold Anchor Selection (Nested Cross-Validation). Select anchors using only a subset of training folds, and validate that anchor set on a completely sequestered fold that played absolutely no role in the selection process.
